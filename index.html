    # Grunt * grunt first released in January 2012. * node 0.0.1 released in May 2009. * npm 1.0.1 released in April 2011 (node at 0.4) * npm 0.0.1 in January 2010 (node at 0.1).
    # Before grunt * Gnu make: Makefile (gruntfile.js, gulpfile.js etc.) * Shell scripts * Custom build scripts written in javascript * Other stuff
    # When not to use Grunt * Task runner, asset pipeline stuff has to be custom written. * Tasks build code by compiling it in a temporary directory. * yeoman angular fullstack generator "best practice" gruntfile, 667 lines long:
    # Gulp * gulp was released 1.5yrs after grunt in July 2013 * pipelines are node streams * plugins are (mostly) stream transforms, 1 source file in 1 destination file out. * The streams are exposed directly to build files.
    # Things Gulp is not so good at. * Streams transforms are built around transforming files 1:1, n:n/1:n/n:1 more difficult. * Concatenating source maps doesn't always work. * Need to use 4 plugins together for incremental rebuilds. * Show screenshot of https://twitter.com/tomdale/status/435463584085049344 * Example from [here](https://github.com/gulpjs/gulp/) PIPE PIPE PIPE POPEYE * Water comes from pipe. * Slowly floods bottom of screen. * Kills cat. * You interact with stream API directly, need to write "pipe" a lot.
    # Broccoli * First release May 2013 * Currently in beta * Plugins take one or more trees (filesystem directories). * Output one or more trees (containing built assets). * "The filesystem is the API"
    # Current issues with Broccoli * Source maps only sometimes working. * Use of filesystem means plugins must clean-up. * Everything sequential, can't parallelise: "only a 40% speed gain". * Brocfile.js files can get verbose.
    # Other pipelines * metalsmith * plumber (no changes in 6+ months). * gobble
    # Problems with Everything * I make a change while the tests are running and: * Let me ignore that for you. (TODO: troll bullet point) * Let me get to that when your tests finish running. * Production source maps. * Need special handling for watching files.
    # sigh * Functional Reactive Programming (sloth hugging kitten) * Like lodash of streams (flatten, reduce etc.) * n:n operations: in sigh each event is an array of files rather than a single file. * Does not use fs except when absolutely necessary. * Source maps from minified/transpiled files back to original sources. * Very neat Assetfile.js syntax. * Only plugin authors have to interact with stream API. * Abstract from sigh.js, array and plugin based tree structure.
    # Uses multiple CPUs/cores * Show sigh being faster than compiling files directly with babel.
